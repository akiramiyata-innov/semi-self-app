# 遠隔接客チャットアプリ テスト計画書

**バージョン**: v0.6.0  
**作成日**: 2026-06-22  
**テスト種別**: 自動テスト（ユニット・インテグレーション・E2E）

---

## 1. テスト概要

### 目的
- 音声認識・翻訳・TTS・リアルタイム通信の正確性を確保
- マルチオペレーション環境での競合制御が正しく機能することを検証
- ログ保存機能の信頼性を確認
- ネットワーク遅延やエラー時の動作を検証

### スコープ
- バックエンド API（翻訳・TTS・ログ）
- Socket.IO サーバー（通信制御・競合制御）
- カスタムフック（音声認識・画面共有）
- ユーティリティ関数（キャッシュ・言語設定）
- 基本的な React コンポーネント（Avatar・TranscriptPanel）

### 対象外（将来のテスト）
- E2E UI テスト（Playwright/Cypress）
- パフォーマンス・負荷テスト
- 外部API（Google Cloud）の実際の呼び出し（モック化）

## 2. テスト層別計画

### 2.1 ユニットテスト

#### 2.1.1 API Route Handlers

**対象ファイル:**
- `app/api/translate/route.ts`
- `app/api/tts/route.ts`
- `app/api/logs/route.ts`
- `app/api/logs/[sessionId]/route.ts`

**テスト項目:**

| # | API | テスト内容 | 期待値 |
|---|-----|----------|--------|
| 1 | POST /api/translate | 正常なリクエストで翻訳結果を返す | ステータス 200、翻訳テキスト |
| 2 | POST /api/translate | キャッシュヒット時は再度APIを呼ばない | キャッシュから即座に返却 |
| 3 | POST /api/translate | 無効な言語コードでエラー | ステータス 400、エラーメッセージ |
| 4 | POST /api/translate | 空のテキスト | ステータス 400 |
| 5 | POST /api/tts | 正常なリクエストで音声生成 | ステータス 200、base64音声データ |
| 6 | POST /api/tts | 無効な言語コード | ステータス 400 |
| 7 | GET /api/logs | ログディレクトリが存在しない場合 | ステータス 200、空配列 |
| 8 | GET /api/logs | 複数セッションのログを日付順で返す | 新しい順にソート |
| 9 | GET /api/logs/[sessionId] | 有効なセッションIDで詳細を返す | ステータス 200、SessionLog オブジェクト |
| 10 | GET /api/logs/[sessionId] | 存在しないセッションID | ステータス 404 |

### 2.2 インテグレーションテスト

#### 2.2.1 Socket.IO サーバー

**対象ファイル:**
- `server/socketServer.ts`

**テスト項目:**

| # | シナリオ | テスト内容 | 期待値 |
|---|---------|----------|--------|
| 1 | 基本フロー | キオスク → call:request 送信 | サーバーが受け取り、スタッフに通知 |
| 2 | 競合制御 | 2人のスタッフが同時に call:answer | 1人だけ接続、もう1人に call:alreadyTaken |
| 3 | 通話中 | speech:user イベント送信 | サーバーが受け取り、スタッフに中継 |
| 4 | 翻訳中継 | 複数言語での発話 | サーバーが翻訳してスタッフに配信 |
| 5 | 通話終了 | call:end イベント | activeSessions から削除、全スタッフに通知 |
| 6 | ログ保存 | 通話終了時 | JSON ファイルがファイルシステムに保存される |
| 7 | 切断時の処理 | クライアント接続切断 | 関連セッションをクリーンアップ |
| 8 | スタッフ在籍管理 | staff:join, staff:leave | 在籍リストが更新される |
| 9 | 複数セッション | 複数のキオスクから同時呼び出し | 各セッションが独立して管理される |
| 10 | 画面共有 | screen:frame イベント | フレームデータが相手側に配信される |

#### 2.2.2 API 連携フロー

**テスト項目:**

| # | フロー | テスト内容 | 期待値 |
|---|-------|----------|--------|
| 1 | 翻訳 → TTS | テキスト翻訳 → 音声生成 | 順序通り処理され、音声データが返る |
| 2 | キャッシュ最適化 | 同じテキスト複数回 | 2回目以降はキャッシュから提供（API呼び出しなし） |
| 3 | ログ保存 → 取得 | セッション終了 → /api/logs で確認 | ログが正しく保存・取得される |
| 4 | エラーハンドリング | 外部API障害時 | グレースフルに失敗、ユーザーに通知 |

### 2.3 ロジックテスト（ビジネスロジック）

**テスト項目:**

| # | ロジック | テスト内容 | 期待値 |
|---|---------|----------|--------|
| 1 | レース条件 | 2人スタッフが同じ呼び出しに応答 | call:alreadyTaken により競合解決 |
| 2 | 音声エコー検出 | キオスク側の音声 == アバターが話した音声 | エコーを除外（マイク自動OFF） |
| 3 | ステータス自動リセット | スタッフのセッション終了 | ステータスが「対応中」から「対応可能」に戻る |
| 4 | マイク自動制御 | スタッフが話している最中 | キオスク側のマイクが自動OFF |
| 5 | 言語自動判定 | キオスクで選択した言語 | 以降の翻訳・TTS に反映される |
| 6 | sessionStorage vs localStorage | 複数キオスク端末タブ | スタッフ名が各タブで独立（sessionStorage） |

### 2.4 エラーハンドリング・エッジケーステスト

**テスト項目:**

| # | シナリオ | テスト内容 | 期待値 |
|---|---------|----------|--------|
| 1 | ネットワーク断線 | 通話中にインターネット断 | 自動再接続を試行 |
| 2 | タイムアウト | API が応答しない | 指定時間後にタイムアウトエラー |
| 3 | 無効なデータ型 | 不正な JSON 送信 | サーバーがエラーを捕捉・ログ出力 |
| 4 | メモリリーク | 長時間実行 | activeSessions が適切にクリーンアップされる |
| 5 | 同時大量接続 | 100個以上のセッション同時生成 | サーバーが正常に処理・管理 |
| 6 | ファイルシステム満杯 | ログ保存時にディスク満杯 | エラーをハンドル・ログ記録 |

## 6. リスク・制限事項

| リスク | 対策 |
|-------|------|
| 外部API（Google）がテスト中に変更 | API仕様をモック化・固定化 |
| Socket.IO のバージョン差異 | サポートバージョンを明記 |
| ブラウザAPI（Web Speech）テスト難度 | jsdom でのモック化 |
| ファイルシステム依存 | temp ディレクトリ使用 |

## 次のステップ

1. **テストコード生成**: Jest テストファイル作成（各レイヤー別）
2. **テスト実行**: `npm run test` で全テスト実行
3. **カバレッジレポート**: `npm run test:coverage` で確認
4. **CI/CD 統合**: GitHub Actions で自動実行
